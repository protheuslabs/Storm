# Operating Principles

These principles should guide product, design, engineering, and policy decisions.

## Route Value To Creators

Storm should minimize unnecessary extraction. Platform fees, reviewer fees, payment processing fees, and dispute reserves should be visible and justified.

## Make Work Small Enough To Execute

Tasks should be scoped so a contributor can understand the expected output, complete it independently, and know how it will be evaluated.

## Define Done Before Work Starts

Every task should have acceptance criteria before it can be claimed. If done cannot be defined, the task is not ready.

## Make Review Accountable

Reviewers influence whether contributors get paid. Their decisions should affect their own reputation, especially when disputes show a review was careless or biased.

## Prefer Transparent Rules Over Private Authority

Storm should avoid hidden decision-making where possible. Participants should understand how claiming, review, payout, reputation, and disputes work.

## Explain Value Allocation

When Storm allocates value, users should be able to see the rule, contribution records, and explanation that produced the result. Hidden value scores should not become a new intermediary.

## Reward Useful Coordination

Some coordination work is real work. Task decomposition, clarification, review, dependency management, and dispute resolution can all create value and may deserve compensation or reputation.

## Preserve Provenance

Storm should track where useful work came from. Ideas, research, decomposition, submissions, and reviews may all become important when later work creates value.

## Keep Reputation Portable In Spirit

Even if reputation starts inside Storm, it should be based on concrete work artifacts, task outcomes, and review history rather than vague ratings.

## Design Against Gaming

The system should assume users may try to farm reputation, collude on reviews, spam low-quality submissions, squat on tasks, or create fake demand.

## Start Narrow

Storm should launch with constrained workflows and a small number of task types before generalizing. A trustworthy narrow workflow is more valuable than a broad but ambiguous marketplace.
