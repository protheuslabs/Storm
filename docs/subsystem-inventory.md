# Subsystem Inventory

This inventory tracks the major Storm subsystems that should stay modular.

The purpose is not to turn Storm into microservices early. The purpose is to know which parts of the system own which authority, which contracts should exist, and which pieces can later be replaced or strengthened by InfRing primitives.

## Priority Levels

- MVP-critical: needed for the first useful Storm loop.
- Early platform: not necessarily required for the first narrow release, but should be designed soon.
- Growth: needed as marketplace scale, applets, groups, or value routing grow.
- Future substrate: likely to depend on mature InfRing or network primitives.

## MVP-Critical Subsystems

### Workflow Orchestration

Owns the canonical state transitions across the core loop:

`Project -> Task -> Claim -> Submit -> Review -> Ledger -> Reputation`

Responsibilities:

- Validate workflow state transitions.
- Prevent invalid transitions.
- Coordinate calls into review, ledger, reputation, notification, and future rights checks.
- Emit workflow events.
- Keep applet UI from becoming workflow authority.

Why it matters:

If workflow logic scatters across controllers, applets, UI, and ledger code, Storm will become hard to audit and harder to migrate to InfRing.

### Identity, Permissions, And Roles

Owns who a participant is allowed to be and what they can do.

Responsibilities:

- User accounts.
- Public contributor identity.
- Private legal/payment identity.
- Project roles.
- Reviewer roles.
- Resolver/admin roles.
- Future group roles.
- Future node identity mapping.

Why it matters:

Storm will need different identities for reputation, payment, legal compliance, privacy, and future network participation.

### Project And Task Core

Owns generic Storm project and task records.

Responsibilities:

- Projects.
- Tasks.
- Claims.
- Submissions.
- Task dependencies.
- Task status.
- Project status.
- Applet attachment points.

Why it matters:

This is the base loop. Applets should extend it, not fork it.

### Review And Dispute Resolution

Owns validation, challenge, and appeal workflows.

Responsibilities:

- Review assignment.
- Review decisions.
- Change requests.
- Rejection evidence.
- Multi-review rules.
- Dispute creation.
- Dispute evidence.
- Resolver decisions.
- Review accuracy signals.

Why it matters:

Review decides whether people get paid. That authority must be explicit, accountable, and auditable.

### Delivery Planning And PM

Owns planning, delivery, and reporting layers that sit above core task execution.

Responsibilities:

- Board-style and table views for active work.
- Milestones, objectives, and release planning.
- Dependency graphs.
- Sprints/cadence controls.
- Custom fields and project metadata templates.
- Status taxonomies and reporting rollups.
- Cross-project portfolio and throughput views (future).

Why it matters:

Task execution is the base loop; delivery planning turns Storm into a true work management layer for teams.

### Value Ledger

Owns economic records.

Responsibilities:

- Ledger entries.
- Fee records.
- Payout allocations.
- Refunds.
- Adjustments.
- Payout holds.
- Reconciliation with future payment provider events.

Why it matters:

The ledger is economic history. Other subsystems can request ledger events, but should not silently mutate economic state.

### Basic Reputation

Owns evidence-based trust signals derived from verified events.

Responsibilities:

- Reputation events.
- Contributor reliability signals.
- Reviewer accuracy signals.
- Dispute outcome effects.
- Domain-specific reputation dimensions.

Why it matters:

Storm should avoid vague ratings. Reputation needs to come from concrete behavior.

### Story Forge Applet

Owns the first domain-specific project experience.

Responsibilities:

- Story project metadata.
- Story task templates.
- Story review rubrics.
- Canon/draft acceptance semantics.
- Story package exports.

Why it matters:

Story Forge proves the applet model without forcing every story concept into Storm core.

## Early Platform Subsystems

### Artifact Storage

Owns durable references to submitted work and project outputs.

Responsibilities:

- Submission artifacts.
- File references.
- Links.
- Story exports.
- Evidence bundles.
- Future code diffs, media, CAD files, datasets, and documents.
- Artifact checksums or content hashes.

Why it matters:

Provenance, rights enforcement, review, receipts, and value routing all depend on stable artifact references.

### Notification And Messaging

Owns participant communication generated by workflow events.

Responsibilities:

- Claim reminders.
- Review requests.
- Dispute notifications.
- Payout updates.
- Rights review alerts.
- Merge proposal updates.
- Project owner/contributor messaging rules.

Why it matters:

Crowdsourced projects fail quietly if participants do not know what needs attention.

### Audit/Event Bus

Owns structured internal events.

Responsibilities:

- Event schemas.
- Event emission rules.
- Audit log projection.
- Event consumers for notifications, reputation, ledger, receipts, and analytics.
- Future receipt mapping.

Why it matters:

Storm needs one event spine, not every subsystem inventing its own shadow history.

### Policy Engine

Owns versioned rules that affect money, rights, access, review, governance, and reputation.

Responsibilities:

- Value policy execution.
- Fee policy.
- Review policy.
- Rights policy.
- Reputation policy.
- Applet merge policy.
- Governance policy.
- Policy versioning.

Why it matters:

Hidden rules recreate the intermediary problem. Versioned policy makes decisions explainable and later receipt-backed.

### Artifact Provenance And Contribution Graph

Owns the relationship between people, work, artifacts, tasks, ideas, reviews, forks, datasets, and downstream value.

Responsibilities:

- Contribution events.
- Provenance links.
- Dependency edges.
- Source lineage.
- Future residual claim inputs.
- Future AI training source records.

Why it matters:

This is the backbone of Storm's value thesis.

## Growth Subsystems

### Payments, Escrow, And Compliance

Owns real money movement and compliance boundaries.

Responsibilities:

- Funded task balances.
- Payment provider abstraction.
- Payout provider abstraction.
- Payout status.
- Escrow-like behavior where legally possible.
- Refunds and chargebacks.
- Tax and compliance records.
- Payout holds from disputes or rights review.

Why it matters:

The ledger models economics; payments move money. They should be connected but not collapsed.

### Fraud, Risk, And Anti-Collusion

Owns detection of behavior that exploits the system.

Responsibilities:

- Review ring detection.
- Duplicate-account risk.
- Task farming.
- Fake work.
- Collusive reviews.
- Suspicious payout patterns.
- Value-policy gaming.
- Rights-enforcement abuse.
- Unauthorized bot or agent use.

Why it matters:

Reputation is not enough. Storm needs a separate risk layer that protects the system without turning every contributor into a suspect.

### Agent Participation And Bot Control

Owns the boundary between authorized automation and undisclosed bots pretending to be humans.

Responsibilities:

- Agent registration.
- Agent authorization scopes.
- Task execution-mode policy.
- Human-only task enforcement.
- Agent-assisted disclosure records.
- Bot-risk signals.
- Agent session records.
- Responsible-user accountability.
- Agent-related reputation and payout effects.

Why it matters:

Storm should welcome useful agents through the right channels, but undisclosed automation can corrupt reputation, review, payouts, governance, and trust.

### Search, Discovery, And Matching

Owns marketplace liquidity and matching.

Responsibilities:

- Project discovery.
- Task discovery.
- Contributor search.
- Reviewer matching.
- Skill indexing.
- Applet-specific search.
- Recommendation and ranking.
- Saved interests.

Why it matters:

Storm can have good tasks and contributors and still fail if they cannot find each other.

### Privacy, Consent, And Data Access

Owns data visibility and consent boundaries.

Responsibilities:

- Private submissions.
- Public profiles.
- Legal/payment identity privacy.
- Dataset inclusion consent.
- AI training consent.
- Rights scanning privacy.
- Role-based data access.
- Export and deletion policies where legally required.

Why it matters:

Storm will handle sensitive work, money, identity, legal records, and training data. Privacy cannot be bolted on later.

### API And Integration Layer

Owns external system boundaries.

Responsibilities:

- Public API.
- Webhooks.
- GitHub integration.
- Figma or design integrations.
- Google Docs or document integrations.
- Payment providers.
- Publishing platforms.
- Patent/search providers.
- InfRing adapters.

Why it matters:

Storm will become more valuable when it can connect to existing work tools without surrendering authority to them.

## Future Substrate Subsystems

### Task Decomposition

Owns conversion from goals into executable tasks.

Near term:

- Manual tasks.
- Applet templates.
- Lightweight backend decomposition helpers.

Long term:

- InfRing task decomposition primitive.

### Versioning And Forks

Owns project lineage, forks, branches, and merge proposals.

Near term:

- Export versions.
- Parent project lineage.
- Simple fork records.

Long term:

- InfRing or shared version/fork subsystem.

### Originality And Rights Enforcement

Owns plagiarism, license, prior-art, and external-rights risk checks.

Near term:

- Contributor attestations.
- Manual disputes.
- Preserved source references.

Long term:

- AI-assisted similarity.
- Rights evidence bundles.
- Receipt-backed rights review.

### AI Training Provenance

Owns dataset, training, output, and AI value traceability.

Near term:

- Clean contribution and artifact provenance.

Long term:

- Dataset manifests.
- Training-run records.
- AI attribution entries.
- Compensation routing.

### Governance And Release Switch

Owns the evolution from Protheus Labs-operated platform to network-governed system.

Responsibilities:

- Governance policy.
- Protocol origin allocation.
- Emergency authority.
- Release criteria.
- Hidden-control audits.
- Network governance.

## Cross-Cutting Rule

Every subsystem should answer:

- What state does it own?
- What events does it emit?
- What policies does it obey?
- What other subsystem contracts does it call?
- What data does it expose to applets?
- What future InfRing primitive might replace or strengthen it?

If a subsystem cannot answer those questions, it is not ready to become a core authority.

## First Build Bias

The first build should focus on:

1. Workflow orchestration.
2. Identity, permissions, and roles.
3. Project and task core.
4. Review and dispute basics.
5. Value ledger.
6. Basic reputation events.
7. Story Forge applet.
8. Artifact storage.
9. Audit/event bus foundation.

Everything else should be shaped by clear interfaces but kept out of the first implementation unless the MVP genuinely needs it.
