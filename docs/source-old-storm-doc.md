## Core Ideas From Source

- Creative ideas have value.
- Positive actions should be compensable.
- Large organizations and governments capture too much value from contributors.
- Crowdsourcing and automation can remove much of the middle layer.
- Contributors should keep roughly 95% of the value created.
- Contribution impact can last over time, so compensation may need to be residual.

## Original Product Shape

Users enter ideas or goals. Other users help build those ideas into something that creates real value. Contributors are rewarded according to how much they helped grow the idea.

The source frames Storm as more than a job marketplace. It is a system for turning goals, ideas, problems, suggestions, research, and real-world tasks into nested units of contribution.

## Containers And Microtasks

The source describes ideas and goals as containers. Users break containers into smaller containers such as:

- Problems.
- What-if prompts.
- Parts.
- Answers.
- Subtasks.

Those are broken down until they become very small tasks that can be matched to users by skill, inclination, and trust.

## Groups And Alliances

The source describes groups as a hybrid between a business and a Discord-style community. Groups may:

- Recruit members.
- Stay open or use member caps.
- Use different power structures, including admins or democracy.
- Seed parts of a project.
- Choose which contributions merge into the main project.
- Maintain multiple iterations.

The older notes also introduce alliances: more durable groups that can emerge from trending problems, suggestions, or public demand. Alliances may use custom governance, check-and-balance systems, and child group seeding.

## Residual Compensation

The source proposes residual compensation for the lifetime of an idea's value.

Key concepts:

- Contribution weight determines compensation.
- Contributions can be diluted downstream as an idea grows.
- A contributor may be compensated when later work builds on their contribution.
- Legacy ideas may have limited residual depth.

This is important to the long-term vision, but it is much more complex than the first MVP's task payout ledger.

## Intellectual Property Protection

The source proposes using AI to categorize ideas by concept and detect plagiarism or unauthorized restarts. Possible penalties include project removal, temporary bans, or future compensation penalties.

The source also mentions:

- Making duplicate accounts hard.
- Limiting full-idea access after the root idea exists.
- Creating provisional patents at a developed stage.

These raise legal, privacy, and trust questions that need careful treatment before implementation.

## Builder Concept

The source describes a builder surface where strong contributors can be invited into later project upgrades. It also suggests connecting external creation tools first, then eventually building Storm-native tools.

This maps to a future project execution environment, not the first MVP.

## Tadpole Business Model

The source describes a self-adapting group/business framework that changes structure based on size, revenue, age, resources, and competition.

The model suggests a group with shared DNA that can seed child groups or businesses, with the parent receiving some royalty. This points toward future organizational templates, group evolution, and residual value between related groups.

## World Eye

World Eye is described as a crowdsourced intelligence network feeding Storm.

Users collect and organize information about real-world problems, events, and trends. Information has a veracity score based on:

- Peer confirmation.
- Anti-edit or image analysis.
- User trustworthiness.

Users can add bounties to solutions, collaborate on research, and store trends in experimental data. If Storm or Bolt creates value from cataloged information, intelligence contributors may be compensated.

## Bolt

Bolt is described as a real-world task bounty network.

Real-world tasks have rewards based on importance. Qualified users receive tasks based on skill and trust. Examples include finding lost people or belongings, escorting someone to a car, or responding to panic signals.

The stated end goal is to crowdsource and automate work usually handled by governments or companies.

This is far outside MVP because it carries significant safety, legal, emergency response, and liability risk.

## Far-Future Vision

The old source includes a much broader future vision involving robotics, fabrication labs, home manufacturing, a currency-like reputation system, space travel, and replacement of major institutions.

This should be treated as inspirational source material rather than near-term product scope.

## Practical Takeaway

The old document expands Storm from a work marketplace into a broader value-coordination system.

Near-term Storm should still start with:

Project -> Task -> Claim -> Submit -> Review -> Ledger -> Reputation.

But the architecture should leave conceptual room for:

- Ideas before projects.
- Containers before tasks.
- Groups before companies.
- Residual value before one-time payments.
- Intelligence and research inputs before project creation.
- Real-world bounties only after the digital workflow is mature and legally safe.
